## What it is and what "good" looks like

### Confirmed by sources
- OpenAI is a managed API surface for text, reasoning, multimodal, image, audio, embeddings, and batch workloads; it is consumed over HTTPS and exposes organization usage and costs APIs for operational and billing visibility. [S1][S2][S3]
- The OpenAI Usage API can return grouped usage details including `input_tokens`, `output_tokens`, `input_cached_tokens`, `input_audio_tokens`, `output_audio_tokens`, `num_model_requests`, and grouping keys such as project, API key, model, and batch. [S1][S2]
- OpenAI pricing is primarily token-based, with distinct rates for input, cached input, and output tokens, so workload mix changes can move spend materially even when request count stays flat. [S3]
- OpenAI rate limits are enforced on both requests and tokens, and the API returns remaining-budget and reset headers that are directly useful for dashboarding and client-side backoff logic. [S4]
- The OpenAI Python SDK exposes request IDs and typed errors, which makes it practical to join client-side logs and traces to provider-facing failures. [S5]
- OpenTelemetry GenAI semantic conventions define a small but high-value metric surface centered on operation duration and token usage, plus spans/attributes for model, provider, finish reasons, errors, and token accounting. [S6][S7][S8]
- Good operational state is low and stable latency for the selected model tier, low retryable error share, predictable token-per-request shape, healthy cache reuse when prompt caching is expected, and spend growth that is explainable by traffic or model mix rather than silent prompt/output inflation. [S2][S3][S4][S6][S7]
- Paging intent in dashboard form: quickly separate provider-side incident, client throttling, prompt/output inflation, and model-mix drift before application teams assume their own code is broken.

### Best-practice inference
- Incident shape 1: **Provider or network degradation.** Signals: latency rises across many services, 5xx or connection errors rise, request IDs cluster around failures. Start in `service-health`.
- Incident shape 2: **Self-inflicted throttling.** Signals: 429 share rises while total request volume or token volume is near known tier ceilings. Start in `rate-limit-headroom`.
- Incident shape 3: **Cost blowout without obvious traffic growth.** Signals: requests are flat but tokens/request, output/input ratio, or model mix shift upward. Start in `spend-model-mix`.
- Dashboard success criteria: one screen should answer whether the problem is latency, failures, budget pressure, or request-shape drift; deeper views should show model, project, API key, and operation ownership boundaries without forcing log spelunking first.

## Key concepts

### Glossary

| Term | Definition | Operational meaning | Dashboard section |
|---|---|---|---|
| Response API request | A single call to OpenAI that creates a model response | Core request unit for latency, errors, and spend attribution | service-health |
| Model | The exact model requested or returned | Drives latency, quality, and price differences | spend-model-mix |
| Operation | The GenAI action type such as chat, response generation, or embeddings | Separates slow inference from cheap utility operations | service-health |
| Input tokens | Tokens sent to OpenAI in the request | Main driver of cost and prompt-size latency impact | tokens-throughput |
| Output tokens | Tokens generated by OpenAI in the response | Main driver of latency and marginal completion cost | tokens-throughput |
| Cached input tokens | Input tokens served from provider-side prompt cache | Key efficiency lever for repeated prefix-heavy workloads | cache-efficiency |
| Audio input tokens | Audio tokens ingested by audio or multimodal workloads | Distinguishes speech-heavy workloads from text-only traffic | tokens-throughput |
| Audio output tokens | Audio tokens generated in model output | Important for voice product cost analysis | tokens-throughput |
| Total tokens | Input plus output token volume | Best broad consumption proxy when request count hides request size | tokens-throughput |
| Tokens per request | Average token footprint of each call | Surges indicate prompt bloat, runaway output, or new workload class | spend-model-mix |
| Output/Input ratio | Output tokens divided by input tokens | Quick signal for verbosity creep and tool-loop amplification | latency-performance |
| Prompt caching | Reuse of cached prompt prefixes to cut latency and cost | Missing cache reuse where expected usually means prompt drift | cache-efficiency |
| Rate limit budget | Remaining request/token allowance before the next reset | Low headroom predicts 429s before customers see hard failures | rate-limit-headroom |
| 429 throttling | Too-many-requests rejection from OpenAI | Usually recoverable, but user-visible if client retries are weak | errors-throttling |
| 5xx provider error | Server-side failure returned by the provider | Strong indicator the problem is external to your application logic | errors-throttling |
| Retryable error | Error type likely to succeed on retry, such as 429 or transient 5xx | Important to separate from hard 4xx request-shape bugs | errors-throttling |
| Project | OpenAI org project boundary | Useful billing and ownership partition | spend-model-mix |
| API key | Credential-level usage boundary | Helps isolate noisy tenants or rogue callers | spend-model-mix |
| Batch request | Request executed through the Batch API | Lower cost but much higher latency tolerance | spend-model-mix |
| Service tier | Provider execution tier or latency tier metadata | Useful for explaining latency and cost differences | service-health |
| Finish reason | Why generation stopped | Distinguishes normal completion from truncation or tool stop | latency-performance |
| Request ID | Provider-issued identifier returned in response headers/SDK | Required for joining failing requests to support tickets and logs | service-health |
| System fingerprint | Provider-side execution fingerprint | Useful for correlating regressions after backend changes | service-health |
| Embedding call | Vector-generation request | Typically high input, low output, different cost/latency shape | tokens-throughput |
| Streaming response | Response delivered incrementally | Helps user-perceived latency, but does not change token spend | latency-performance |

### Concept Map
- Client application -> sends -> OpenAI HTTPS API request (why: every latency, error, and token signal starts here)
- OpenAI request -> selects -> model (why: model choice changes latency, cost, and rate-limit headroom)
- OpenAI request -> consumes -> input tokens (why: larger prompts increase cost and often latency)
- OpenAI response -> produces -> output tokens (why: generated volume often dominates latency and spend)
- Input tokens -> may hit -> provider prompt cache (why: cached prefixes lower both cost and latency)
- Prompt cache miss -> increases -> uncached input tokens (why: silently raises cost even if request count is flat)
- Output tokens -> increases -> wall-clock latency (why: generation time scales with produced content)
- Batch API -> trades -> lower cost for slower completion window (why: not every request should be compared to online latency baselines)
- Rate limit budget -> constrains -> request throughput (why: 429s appear before demand actually falls)
- Token limit budget -> constrains -> large prompts and long outputs (why: request count can look safe while token budget is exhausted)
- 429 responses -> trigger -> client retry and backoff behavior (why: bad retry policy turns throttling into user-visible outage)
- 5xx responses -> imply -> provider or network instability (why: usually not fixed by prompt changes)
- Client logs -> record -> request ID (why: request ID is the bridge to support/debugging)
- SDK traces -> enrich -> model, operation, error type, token attributes (why: gives ownership and root-cause dimensions)
- Usage API -> aggregates -> tokens and request counts by time bucket (why: good for spend and capacity trend dashboards)
- Costs API -> aggregates -> dollar spend by day, project, and line item (why: authoritative financial surface)
- Project -> owns -> API keys (why: blast radius and chargeback often follow project boundaries)
- API key -> originates -> traffic from one workload or tenant (why: isolates noisy callers)
- Service/team/env -> map -> owning application boundary (why: responders need organizational routing)
- Model mix -> changes -> latency and spend posture (why: switching to larger models shifts both quickly)
- Streaming -> improves -> time to first visible output (why: user experience may improve even if total generation time is unchanged)
- Finish reason `length` -> hints at -> truncation or low max token settings (why: short outputs may be config error, not success)
- Error type -> separates -> bad request vs retryable provider failure (why: fixes differ completely)
- Audio workloads -> change -> token accounting mix (why: text-only baselines become misleading)
- Embeddings -> have -> different latency and token shape than text generation (why: sections need operation-aware breakdowns)
- Prompt template drift -> reduces -> cache reuse (why: tiny prefix changes can erase expected savings)
- Provider incident -> first raises -> latency and error rate across many callers (why: cross-service correlation matters more than single app noise)
- Cost spike -> may come from -> larger outputs before request count moves (why: token ratios are more valuable than raw requests)
- Ownership fields -> bound -> safe top-N group-bys (why: model/project/key dimensions are useful, raw prompt/user fields are not)

## Golden signals

### Confirmed by sources
| Signal | What it means for OpenAI | Typical degradation causes | Best telemetry sources | What people page on | Section questions |
|---|---|---|---|---|---|
| Traffic | Request and token volume sent to the API | Product launch, retry storm, prompt inflation, batch release | Usage API request/token buckets; `gen_ai.client.token.usage` [S1][S2][S6] | Sudden jump in tokens or requests without planned event | Is traffic normal? Is growth request-count driven or token-shape driven? |
| Errors | 4xx/429/5xx and SDK/network failures | Bad auth, malformed requests, quota/rate limits, provider incident | SDK errors, HTTP status, traces with `error.type`, logs with request IDs [S4][S5][S7] | Sustained 429 share or broad 5xx/connection failures | Are failures self-inflicted, provider-side, or bad-request regressions? |
| Latency | End-to-end time to receive a response or streamed output | Model choice, larger outputs, provider congestion, network path issues | `gen_ai.client.operation.duration`, streaming spans, latency docs [S6][S9] | P95 latency rising across many services or models | Is it the model, output size, or a broader provider issue? |
| Saturation | Near-exhaustion of request/token budgets or client retry pools | Tier limits, burst traffic, long outputs consuming token budget | Rate-limit headers; usage/token trend; 429s [S4] | Remaining headroom collapses or 429s persist | Are we close to token/request ceilings? Which workload is consuming the budget? |

### Best-practice inference
- For OpenAI, token volume and model mix are more operationally important than generic CPU or host metrics because the external dependency is priced and throttled primarily on tokens and requests.
- Prompt-cache share belongs beside latency and cost, not as a niche optimization metric, because lost cache reuse can simultaneously worsen both.

## Telemetry sources

### Confirmed by sources
| Source type | How collected | What it provides | Pros/cons | Common pitfalls |
|---|---|---|---|---|
| OpenAI Usage API | Poll `/v1/organization/usage/completions` with admin key | Bucketed tokens, request counts, model/project/API-key/batch breakdowns | Best source for usage trends and allocation; not real-time per-request | Can differ slightly from reconciled costs; requires admin credentials [S1][S2] |
| OpenAI Costs API | Poll `/v1/organization/costs` | Bucketed spend by project and line item | Best financial source of truth | Daily granularity only; not ideal for minute-by-minute incident triage [S1][S2] |
| OpenAI response headers | Capture `x-ratelimit-*` and `x-request-id` in client instrumentation/logs | Remaining budget, reset timers, request correlation IDs | Very actionable for throttling and debugging | Often lost unless SDK middleware or raw response access is enabled [S4][S5] |
| OpenAI SDK logs/errors | Application-side logging and exception capture | Typed errors, request IDs, retry context | Best for correlating failures to code paths | Must avoid logging prompts or sensitive payloads [S5] |
| OpenTelemetry GenAI metrics | Instrument app/agent SDK with GenAI semantic conventions | Operation duration and token usage metrics | Standardized and vendor-portable | Small metric surface; many useful dimensions live on spans, not metrics [S6] |
| OpenTelemetry GenAI spans | Emit spans/attributes for model, provider, finish reason, token usage, errors | Rich dimensions and root-cause context | Best troubleshooting surface | Attribute sets are still in development and may vary by library/version [S7][S8] |
| OpenAI status page | External incident/status feed | Aggregate provider outage context | Helpful sanity check during widespread incidents | Aggregate availability may not match your exact tier/model mix [S10] |

### Best-practice inference
- "No data" from Usage or Costs APIs usually means the poller/exporter is absent or under-scoped, not that OpenAI traffic is truly zero.
- "No data" from GenAI metrics often means instrumentation is missing while the application is still calling OpenAI successfully.
- When both Usage API rollups and request-level tracing are available, use the API rollups for spend truth and traces for incident explanation.

## Log intelligence (Stage 4 handoff)

### Confirmed by sources
1. OpenAI’s SDK/docs confirm request IDs are emitted from the `x-request-id` response header and are intended for debugging and support correlation. [S5]
2. The Python SDK exposes typed errors, retries, and raw-response header access, which means application logs can reliably capture status code, request ID, retry count, and model when instrumented. [S5]
3. The API surface is HTTPS JSON over `/v1/...` endpoints, so gateway/proxy/access logs are a realistic Stage 4 source even when provider-side logs are not available. [S1][S3][S5]

| Source | Access path | Typical format | Structured vs unstructured | Evidence |
|---|---|---|---|---|
| Application logs around SDK calls | Service stdout / app logger | JSON or key-value app logs | Usually structured if app logger is structured | [S5] |
| API gateway / ingress logs | Reverse proxy, edge gateway, service mesh | HTTP access logs | Mixed | [S5] |
| OpenTelemetry traces exported as logs/events | OTEL collector or app instrumentation | Structured telemetry events | Structured | [S7][S8] |

### Best-practice inference
**Known log formats**
- Structured app log example: `{"msg":"openai request failed","model":"gpt-5","status_code":429,"request_id":"req_abc","retry_count":2,"duration_ms":1834}`
  Shape notes: JSON object; timestamp/level/logger wrapper depends on application logger; safe target fields are model, request_id, status_code, error_type, retry_count, duration_ms.
- Access-log example: `2026-04-10T11:02:33Z api-gateway GET /v1/responses 429 upstream=api.openai.com req_id=req_abc duration_ms=1834`
  Shape notes: line-oriented text; whitespace-delimited or key-value; request ID may be propagated from upstream response header into gateway log.

**Candidate query filters for Stage 4**
- Precise: `context.server.address:api.openai.com OR url.full:*api.openai.com*`
  Rationale: catches calls truly going to OpenAI and avoids unrelated AI-adjacent logs.
  Risk: misses logs where only endpoint path or request_id is recorded.
- Fallback: `message:*openai* OR message:*api.openai.com* OR message:*req_*`
  Rationale: useful when logs are semi-structured and only mention provider name or request id.
  Risk: higher false-positive rate from config, docs, or unrelated application messages.

**Attribute mapping hints**

| Raw field | Suggested Tsuga key | Confidence | Notes |
|---|---|---|---|
| `request_id` / `x-request-id` | `request.id` | High | Most important support/debug join key |
| `model` | `context.gen_ai.request.model` | High | Normalize requested model name |
| `response_model` | `context.gen_ai.response.model` | Medium | Use when different from requested alias |
| `status_code` | `http.response.status_code` | High | Needed for 429/5xx routing |
| `error_type` | `context.error.type` | High | Prefer normalized SDK error family |
| `retry_count` | `openai.retry.count` | Medium | Useful custom key if retries are logged |
| `duration_ms` | `event.duration_ms` | Medium | Prefer numeric duration field if present |
| `project_id` | `context.openai.project.id` | Medium | May only exist in app-side structured logs |
| `api_key_id` | `context.openai.api_key_id` | Medium | Capture only redacted/ID-safe form |
| `service_tier` | `context.openai.response.service_tier` | Low | Only if explicitly logged |

**Parsing risks**
- Request IDs may only appear on failure paths unless raw response headers are always captured.
- Some SDKs log error type but not HTTP status unless the caller adds middleware.
- Reverse proxies may log the upstream host but not model or token details.
- Prompt/content logging must remain off; Stage 4 should parse metadata, not payload text.
- Mixed structured/unstructured logs across services may require split processors by logger name or route.

## Caveats and footguns
- **[service-health]** `gen_ai.client.operation.duration` is the safest confirmed latency metric, but it may include client-side transport time and retries, not only provider compute time. (S6, S7)
- **[latency-performance]** Output tokens often dominate latency; blaming prompt size first can send responders down the wrong path. (S9)
- **[tokens-throughput]** Request count alone is a weak OpenAI KPI because a small number of huge prompts can consume more budget than many tiny requests. (S1, S2, S3)
- **[tokens-throughput]** `gen_ai.client.token.usage` is a compact metric family, but input vs output splits may require a token-type dimension that Stage 2 must verify in Tsuga. (S6)
- **[errors-throttling]** 429s are usually a capacity-policy signal, not a provider outage signal. Treat them differently from 5xx. (S4, S5)
- **[errors-throttling]** Bad-request 4xx spikes can come from deploy-time schema/prompt regressions; auto-retrying them only amplifies waste. (S5)
- **[rate-limit-headroom]** Rate-limit headers are per-response and easy to lose if the client library is wrapped without raw-header capture. (S4, S5)
- **[rate-limit-headroom]** Request and token ceilings can disagree; low token headroom with healthy request headroom is common on long-context workloads. (S4)
- **[cache-efficiency]** Cached input is priced differently from uncached input; losing cache reuse can spike spend even with stable traffic. (S3)
- **[cache-efficiency]** Prompt caching depends on stable shared prefixes; minor template drift can erase the benefit without obvious application change. (S9)
- **[spend-model-mix]** Cost spikes are often model-mix changes, not traffic spikes. Always check model distribution before assuming abuse. (S2, S3)
- **[spend-model-mix]** Costs API is better for invoice-truth than the Usage API; they may not reconcile perfectly bucket-by-bucket. (S1)
- **[service-health]** Aggregate provider status can look green while one model family or tier is degraded for your workload. (S10)
- **[service-health, errors-throttling]** Request IDs should be logged on both success and failure paths; if only failures capture them, correlation during latency-only incidents becomes much harder. (S5)
- **[latency-performance]** Streaming improves user-perceived latency but not necessarily full-response completion time or spend. (S9)
- **[tokens-throughput, spend-model-mix]** Audio token fields can materially alter workload cost shape; text-only assumptions break on multimodal products. (S1, S2)
- **[service-health]** OpenTelemetry GenAI semantic conventions are still marked development, so attribute names can vary by library version. (S6, S7, S8)
- **[tokens-throughput]** Spans contain many useful token attributes, but Stage 3 dashboards should not assume every attribute is also a metric dimension. (S7)
- **[cache-efficiency]** A low cached-token share is not always bad; one-shot or user-unique prompts may have no expected reuse. (Inference)
- **[rate-limit-headroom]** If you only ingest Usage API buckets, you will see throttling after the fact, not impending headroom collapse. Pair rollups with header telemetry where possible. (Inference)
- **[errors-throttling]** Grouping by raw API key IDs can be high-signal for abuse, but exposing too many keys in charts quickly becomes unreadable. Prefer top-list over timeseries. (Inference)
- **[spend-model-mix]** Cost per request can look healthy while cost per useful outcome worsens; keep prompt/output efficiency widgets nearby. (Inference)
- **[service-health]** `context.server.address` may always be `api.openai.com`; if so, it is a filter, not a diagnostic dimension. (Inference)
